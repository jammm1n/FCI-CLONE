# Decision Matrix

## Purpose
Compressed decision rules extracted from case outcomes and mentoring sessions. The AI MUST scan this table at Step 21 (Conclusion & Recommendation) before finalizing any retain/offboard/escalate decision. For each row where the Scenario Pattern matches the current case, state: "Note: Decision Matrix Rule #[X] applies — [Rule]." If multiple rows match and conflict, flag both to the user for a decision.

Full narrative context for each rule is available in case-decision-archive.md under the matching entry number. Consult the archive only when disambiguation is needed or the investigator requests more context.

## Maintenance Protocol
When adding a new lesson:
1. Write the full narrative entry in case-decision-archive.md with the next number
2. Add one row to this matrix with the same number
3. If the new rule SUPERSEDES an existing row, mark the old row as [SUPERSEDED by #X] — do not delete

When the matrix exceeds 50 rows, add category headers within the table (Offboarding, RFI, Escalation, Elliptic, Device, Corporate, etc.).

> **Anti-Fragmentation Rule:** Decision logic (when to retain, offboard, send RFI, escalate) lives in decision-matrix.md only. Step documents contain procedural logic (how to execute) and step-specific analytical frameworks. The system prompt contains behavioral rules and operational lessons. When a new lesson contains both a decision rule and a procedural instruction, split them: the decision goes in the matrix, the procedure goes in the relevant step document, and each references the other by rule number or step number. Never restate the same decision rule in full in multiple documents — use one-line pointers with the rule number.
